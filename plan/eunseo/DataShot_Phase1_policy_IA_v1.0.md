# DataShot 정책서 (Phase 1)

본 정책서는 ReadySet 내 독립 솔루션인 **DataShot(데이터샷)**의 운영 원칙, 처리 방식, 데이터 관리 기준을 정의한다.

## 1. 솔루션 정의

### 1.1 DataShot 정의

- DataShot은 ReadySet 내 독립 솔루션으로, 사용자가 설정한 조건에 따라 내부 캠페인 성과 데이터를 추출하고, 결과를 데이터셋 단위로 저장/관리함.
- “원하는 데이터 집합을 한 컷 선명하게 포착하고, 목표를 정확히 겨냥해 추출한다”는 뜻을 이름에 담음.

### 1.2 DataShot 구조

```
[ adly 메인 로그인 ]
      │
      ▼
[ ReadySet 플랫폼 진입 ]
      │
      ▼
[ Slotboard ] (접근 권한이 있는 Slot 목록)
      │
      ▼
[ Slot 선택 ] (광고주 단위)
      │
      ├─ Reach Caster
      ├─ Budget Optimizer
      ├─ Ad Curator
      └─ DataShot
          │
          ▼
[ 데이터셋 목록 ]
          ├─ 데이터셋 생성 (조회 조건 설정 → 데이터 추출)
          └─ 데이터셋 상세 (메타데이터 / 요약 차트 및 통계 / Raw 데이터 / AnXer AI 챗봇)
```

### 1.3 Dataset (데이터셋) 정의

- 데이터셋은 조회 조건과 추출 시점의 결과를 묶어 관리하는 스냅샷 기반 자산임
- 추출 요청 시점에 파일을 생성하여 고정 저장하며, 이후 상세 조회 시에는 저장된 파일을 불러와 화면에 표시함.
- 스냅샷의 의미 = 추출 시점에 생성된 파일이 이후 변경되지 않는 성질. 즉, 원천 DB 데이터가 변경 되더라도 기존 데이터셋 파일은 유지됨.

<details>
<summary><strong>데이터셋 주요 속성(참고)</strong></summary>

| 구분 | 속성명 | 필수 여부 | 설명 |
|---|---|---|---|

| 기본 정보 | 설명 | 선택 | 데이터셋에 대한 부연 설명 |
| 조회 범위 | 조회 기간 | 필수 | `yy-mm ~ yy-mm` 월별 데이터로 정의 |

| 생성 정보 | 생성자 | 필수 | 데이터셋 생성 요청자 |
| 생성 정보 | 생성일시 | 필수 | 데이터셋 생성 요청 시점 |

</details>

### 1.4 핵심 원칙

| 원칙 | 내용 |
|---|---|
| 불변성 | 생성 완료된 데이터셋의 조회 조건과 결과 값은 수정 불가. 복사 기능을 통해 신규 생성하여야 함 |
| 식별성 | 모든 데이터셋은 고유 ID를 가지며 생성 시점의 메타데이터(생성자, 생성일시, 조회 조건)를 영구 보존함 |
| 권한 상속 | 데이터셋은 상위 Slot의 접근 권한을 자동 상속 받음. 별도 권한 설정 없음. |

## 2. 사용자 및 권한

### 2.1 ReadySet > Slot 기반 데이터 격리

- DataShot은 Slot 접근 권한을 그대로 상속하며, 별도 권한 체계를 추가로 운영하지 않음
- 데이터 격리는 Slot 단위로 유지되어, Slot 간 데이터 상호 접근 불가함
- 데이터 격리 세부 기준은 ReadySet 정책서를 준용함

### 2.2 역할별 기능 권한

| 기능 | 관리자(Admin) | 마케터(Marketer) | 광고주(Client) | 대행사(Agency) | 정의 |
|---|---|---|---|---|---|
| Create | O | O | X | X | 조회 조건 입력 후 데이터셋 생성 요청 |
| Read | O | O | O | O | 데이터셋 목록/상세에서 결과 조회 (Raw 데이터, Export 내보내기) |
| Update | X | X | X | X | - |
| Delete | O | O | X | X | 데이터셋 삭제. 삭제된 데이터셋 복구 불가. |
| Copy | O | O | X | X | 기존 데이터셋 조회 조건을 상속하여 생성 화면으로 진입, 일부 조건 수정하여 신규 생성 가능 |
| Move | O | O | X | X | 동일 광고주 ID Slot 간 이동 |

## 3. 사용자 흐름

### 3.1 데이터셋 생성

- 데이터셋 생성은 사용자가 설정한 조회 조건을 기반으로 해당 시점의 집계 결과를 추출하여 스냅샷 형태로 저장하는 기능임

```
[데이터셋 생성 버튼 클릭]  ← Admin, Marketer만 접근 가능
        │
        ▼
[기본 정보 입력]
 └─ 데이터셋명 (자동 입력, 수정 가능) / 설명 (선택)
        │
        ▼
[조회 조건 선택]
 ├─ 매체 (필수, 단일 선택)
 ├─ 업종 (필수, 다중 선택)
 ├─ 조회 기간 (필수, 최대 24개월)
 ├─ 광고상품 (필수, 다중 선택)
 ├─ 지표 (필수, 다중 선택)
 └─ 타겟팅 옵션 (선택)
        │
        ▼
[Sample Data 미리보기 확인 — 5행]
        │
        ▼
['데이터 추출' 버튼 클릭]  ← 이 시점에 파일 생성 Job 시작
        │
        ▼
[데이터셋 목록에 즉시 등록 — 상태: 대기중]
        │
        ▼   (비동기 처리 — UI와 Job 독립 진행)
[파일 추출 및 생성 — 상태: 추출중]
 └─ Full 파일 + Preview 파일 동시 생성·저장
        │
     ┌──┴──┐
   성공      실패
    │        │
 [완료]   [실패 — 재시도 없음]
            └─ Copy 또는 신규 생성으로 대체
```

### 3.2 데이터셋 조회

- 데이터셋 조회는 생성 완료된 데이터셋의 메타정보, 차트, 요약 통계 및 저장된 결과 데이터를 확인하는 기능임
- 상태가 `완료`인 데이터셋에 한해 조회가 가능

```
[데이터셋 목록]  ← 전 역할 접근 가능
 ├─ 검색 / 정렬 / 페이지네이션
 └─ 데이터셋 카드 클릭
        │
        ├─ [완료 상태]
        │       ▼
        │   [데이터셋 상세]
        │    ├─ 헤더: 조회 조건 + 메타데이터
        │    ├─ 차트 및 통계 요약
        │    └─ Raw 데이터 테이블
        │
        └─ [만료 상태]
                ▼
            "만료된 데이터셋입니다.
             동일한 조회 조건으로 재추출하려면
             데이터셋을 복사하여 새로 생성해 주세요." 안내
                ▼
            Copy → 조건 상속 → 신규 생성
```

### 3.3 Export 내보내기

- 파일 내보내기는 저장된 전체 결과 데이터를 CSV 파일 형태로 다운로드하는 기능임

```
[상세 화면에서 CSV 다운로드 버튼 클릭]
        │
        ▼
[Full 파일 다운로드]
```

- 내보내기 형식 : CSV  
- 파일명 : `[추출자ID]_[추출일시].csv` (강제 삽입, 변경 불가)  
- 화면의 필터 및 정렬 상태와 무관하게 Full 파일 전체 행 다운로드  
- Export 이후 데이터 가공 책임은 사용자에게 있음  

### 3.4 데이터셋 이동

- 데이터셋 이동은 기존 데이터셋의 스냅샷 결과와 메타 정보는 유지한 채, 소속 Slot만 변경하는 기능임

#### 3.4.1 이동 범위
- 동일 광고주 ID를 가진 Slot 간 이동만 가능

#### 3.4.2 메타 정보 유지 원칙
- 메타 정보(Dataset ID, 생성자, 생성일시, 조회 조건)는 변경되지 않음
- 상위 Slot 정보만 변경됨

#### 3.4.3 권한 제약
- Admin, Marketer만 이동 가능
- Client, Agency는 이동 기능 사용 불가

## 4. 처리 구조 및 파일 생성 정책

### 4.1 비동기 처리

- DataShot은 비동기(Asynchronous) 처리를 기본 방식으로 채택함.
- 추출 요청 시 데이터셋이 목록에 즉시 등록되며, 파일 생성은 백그라운드에서 수행됨.

#### 4.1.1 비동기 처리 채택 사유

- 현재 성능 최적화 목적이 아닌, 미래 확장성을 고려한 구조적 선제 설계 결정
- 현재 데이터 규모 기준 동기 처리도 가능하나, 아래 확장 시나리오에 선제 대응하기 위한 비동기 구조 채택

| 확장 시나리오 | 예상 영향 |
|---|---|
| 타겟팅 옵션(성 연령 데모/ 국가 등) 추가 | Row 수 배수 증가 |
| 동일 Slot 내 다수 동시 추출 요청 | 처리 지연 및 충돌 위험 |

따라서, Phase 1부터 비동기 기반으로 표준 흐름을 고정하여, 이후 확장 시 사용자 경험과 운영 구조의 변경 비용을 최소화함  
(단, 개발 중 크리티컬한 이슈 발생 시 추출 시간이 5초 이내일 경우, 동기 전환 검토 가능)

#### 4.1.2 동시 요청 제한 (확인 필요)

- Slot 당 동시 처리 가능한 데이터셋 추출 건수는 N건으로 제한
- 초과 요청은 자동으로 “대기중” 상태로 등록되며, 선행 작업 완료 시 순차적으로 처리
- 서비스 안정성 확보를 목적으로 함

### 4.2 데이터셋 상태 정의

상태 전환은 작업 성공/실패/만료에 의해 자동 변경되며, 사용자에 의해 수동 변경 불가

| 상태 | 설명 | 전환 조건 |
|---|---|---|
| 대기중 | 추출 요청이 접수된 상태 | ‘데이터 추출’ 버튼 클릭 즉시 |
| 추출중 | 파일 추출 및 생성이 진행 중인 상태 | 작업 처리 시작 시 |
| 완료 | Full/Preview 파일 생성이 정상 완료된 상태 | 두 파일 모두 생성 완료 시 |
| 실패 | 오류로 인해 파일 생성이 중단된 상태 | 추출, 파일 생성 중 오류 발생 시 |
| 만료 | 보관 기간 경과로 파일이 삭제된 상태 (메타 데이터 및 조회조건은 유지) | 생성일 기준 1년 경과 시 |

### 4.3 파일 생성 및 화면 표시 정책

#### 4.3.1 생성 시점
- 사용자가 ‘데이터 추출’ 버튼을 클릭하는 시점에 파일 생성

#### 4.3.2 파일 생성 구조
- Full/ Preview 파일은 추출 완료 시 동시에 생성 및 저장됨
- Preview 파일은 기본 정렬 기준은 상세 기획 단계에서 확정

| 구분 | 설명 | 용도 |
|---|---|---|
| Full 파일 | 조회 조건에 해당하는 전체 행 포함 | CSV Export 소스, 차트 및 통계 요약 기준 |
| Preview 파일 | 기본 정렬 적용 후 상위 5,000행 포함 | Raw 데이터 테이블 화면 표시 소스 |

#### 4.3.3 화면 표시 정책
- Raw 데이터 테이블은 Preview 파일 기준(최대 5,000행)으로 표시
- 전체 행 수가 5,000을 초과하는 경우, 사용자 안내 메시지 노출  
  - 예) “전체 N행 중 5,000행만 표시됩니다. 전체 데이터는 CSV 다운로드로 확인하세요.”
- 차트 및 통계 요약은 Full 파일(전체 행)을 기준으로 산출

> **차트/요약 제공 원칙**
> - 차트/통계는 “Full 파일 기준(전체 데이터)”으로 산출됨 (Raw 테이블의 화면 필터/정렬과 독립)
> - 제공할 차트 유형과 구성은 상세 기획 단계에서 확정

#### 4.3.4 타임아웃 유형

| 구분 | 언제 | 처리 방식 |
|---|---|---|
| 화면(UI) 타임아웃 | 생성 요청 후 화면이 60초 이상 응답하지 않을 때 | 화면만 종료. “목록에서 상태를 확인하세요” 안내 |
| 작업(Job) 타임아웃 | 작업 에러 발생 시 | 작업 강제 종료 → 상태: 실패 |

- 화면 타임아웃이 발생해도 추출 작업은 계속 진행됨, 목록에서 상태 확인
- 작업 타임아웃으로 실패 처리된 경우, 복사 또는 신규 생성으로 재시도 가능 (별도 재시도 버튼은 두지 않음)

## 5. 데이터 구성 및 품질 정책

### 5.1 데이터 기준 시점

- DB 업데이트 : 전월까지 집계 완료된 데이터를 매월 5일에 반영  
  - 당월 데이터 및 실시간 성과 반영하지 않음
- 매체사 어드민 오차 : 매체사 보정 로직에 따라 어드민 데이터와 수치 오차 발생 가능
- 보정 고지 : 원천 데이터 업데이트 발생 시, 플랫폼 내 알림을 통해 보정 범위 및 일시 안내
- 기준일 표기 : 조회 화면에 DB 반영 기준일 명시

### 5.2 제공 범위

조회 조건 생성 페이지는 데이터셋의 ‘범위’를 결정하는 단계로, 데이터 집계 구조 및 해석에 영향을 미치는 조건까지 제공함

| 항목 | 필수 여부 | 정책 |
|---|---|---|
| 매체 | 필수 (단일 선택) | 선택 매체에 따라 광고상품/지표/타겟팅 옵션 선택지가 동적으로 결정됨<br>- 대상 매체: Google Ads, Meta, 모먼트, GFA, NOSP, 틱톡 |
| 업종 | 필수 (다중 선택 가능) | 업종(대/중/소) 기반 세분화 조회 제공 (업종 단계 선택은 집계 단위 조정을 위한 옵션) |
| 조회 기간 | 필수 (`yy-mm ~ yy-mm`) | 월 단위 선택을 기본으로 함 |
| 광고상품 | 필수 (다중 선택 가능) | 매체별 상이한 항목 제공 |
| 지표 | 필수 (다중 선택 가능) | 집계 지표 + 효율 지표 제공 |
| 타겟팅 옵션 | 선택 | 집계 단위로서 제공하며, 선택 가능한 옵션은 매체별로 상이함 |

### 5.3 데이터 구성 단위 (Row Depth 기준)

데이터셋은 아래 컬럼 조합의 **최하위 depth**를 기준으로 행을 구성함

| 기간(yy-mm) | 매체 | 업종(대) | 업종(중) | 업종(소) | 광고상품 옵션_1 | 광고상품 옵션_2 | 타겟팅 옵션 |
|---|---|---|---|---|---|---|---|

- 광고상품 옵션의 항목과 개수는 매체별로 상이
- 상위 단위로 재집계하는 기능은 제공하지 않음
- 집계 구조는 데이터셋 생성 후 변경 불가. 다른 기준이 필요하면 복사 후 신규 생성해야 함.

### 5.4 조회조건 결합 원칙

| 결합 유형 | 적용 로직 | 예시 |
|---|---|---|
| 항목 간 (Cross-category) | AND (교집합) | 매체=Meta AND 업종=금융 |
| 항목 내 (In-category) | OR (합집합) | 업종=금융 OR 자동차 |

- 지원하지 않는 조건: `NOT IN` / `!=` 등 부정 조건, 수치형 지표 기반 필터 (예: CTR > 3%)

### 5.5 컬럼 정의

Raw 데이터 테이블과 csv 는 동일한 컬럼 구조(Meta → Metrics 순)를 공유함

| 구분 | 대상 |
|---|---|
| Meta | 매체, 업종, 광고상품, 타겟팅 옵션 |
| 집계 지표 | 원천 데이터 기준 집계된 절대값. (예: 노출수, 클릭수, 광고비) |
| 효율 지표 | 집계 지표 기반 사전 정의 계산식으로 산출된 값. (예: CPC, CPM, CPV, CTR, VTR)<br>계산 결과 값만 제공. 계산식 수정 및 사용자 정의 계산 미지원 |

- 값 처리 기준  
  - 반올림: 모든 수치 지표는 소수점 셋째 자리에서 반올림  
  - Null 처리: Bridge 수집 데이터 특성상 일부 항목은 Null과 0을 구분하지 못함 → 정책적으로 0으로 처리. 해석 시 주의 필요  

### 5.6 조회 결과 0건 예방 정책

#### 5.6.1 정책 배경
- 배경: 사용자가 설정한 조건 조합에 따라 결과가 0건으로 반환될 가능성이 존재함. 모든 조건 조합에 대한 결과 건수를 사전에 전수 집계하여 관리하는 방식은 조건 케이스가 많아 현실적 적용이 어려움
- 목적: 0건 발생으로 인한 사용자의 조회조건 재설정 부담을 줄이고자 함

#### 5.6.2 적용 방식
- 선택지 제한 방식 (UI 화면 레벨 컨트롤)  
  - 상위 조건(예: 매체) 선택 시, 해당 매체 내 실제 성과 데이터가 존재하는 하위 조건(상품, 업종 등)만 활성화하여 노출  
  - 데이터가 존재하지 않는 값은 선택지에서 제외하여 불필요한 조합 생성을 사전에 방지  
- Sample Data 미리보기 제공  
  - 생성 페이지 하단에 컬럼 구조 + Sample Data 5행 제공  
  - 사용자는 선택한 조건이 의도한 데이터 범위가 맞는지, 실제 데이터가 존재하는지 시각적으로 확인할 수 있음  

#### 5.6.3 처리 원칙
- 참조 한정: 샘플 데이터는 존재 여부 및 구조 확인을 위한 참고용 정보이며, 전체 Row 수를 보장하지 않음
- 보증 범위: 본 정책은 0건 발생 가능성을 낮추기 위한 예방 장치이며, 데이터셋 생성 시점의 원천 DB 상태에 따라 결과는 변동 가능함
- 최종 의사결정: 시스템이 제공하는 사전 제어 및 미리보기를 기반으로, 데이터셋 생성 여부에 대한 최종 판단은 사용자에게 있음

## 6. 데이터 보관 및 보안 정책

### 6.1 데이터 보관

데이터셋은 생성일 기준 1년 보관 후, 만료 처리함

#### 6.1.1 만료 처리 기준

| 구분 | 처리 내용 |
|---|---|
| Full 파일 | 삭제 |
| Preview 파일 | 삭제 |
| 조회 조건 | 영구 보존 |
| 메타데이터 | 영구 보존 |
| 상태 | 완료 → 만료(Expired)로 변경 |

#### 6.1.2 Slot 당 쿼터

결과 파일이 존재하는 유효 데이터셋 기준으로 산정

- 단일 Slot당 유효 데이터셋 최대 100개 보유 가능
- 만료(Expired) 데이터셋은 Quota 산정에서 제외
- 삭제(Delete)된 데이터셋은 즉시 Quota에서 차감
- 100개 도달 시 신규 생성 불가. 기존 데이터셋 삭제 후 생성 가능

### 6.2 로그 기록 및 추출 보안

데이터 보안 사고 방지와 정합성 문의 대응을 위해 상세 행위 로그를 관리함

#### 6.2.1 로그 기록 및 관리
- 기록 항목: 데이터셋 ID, 조회 시점, 계정 ID, 접근 IP, 다운로드 여부, 추출된 Row 수
- 관리 원칙: 기록된 로그는 수정 및 삭제가 불가능하며, 최소 12개월간 보관
- 적용 범위: Phase 1 에서는 백엔드 단 로그 적재만 수행, 별도의 로그 조회 화면은 제공하지 않음  
  - 어드민 개발 시 포함하여 진행 예정

#### 6.2.2 데이터 반출(Export) 보안
- 추적 가능성 확보: CSV Export 시 파일명에 `[추출자ID_추출일시]` 형식을 강제로 삽입하여 외부 반출 시 추적 근거를 마련함
- 개인정보 비식별화: Raw 데이터 내 개인 식별 가능성이 있는 브랜드/캠페인 정보 등은 미노출 처리를 원칙으로 함
